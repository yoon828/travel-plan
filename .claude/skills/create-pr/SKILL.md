# GitHub PR 생성 Skill

## 역할
현재 브랜치의 변경 사항을 분석해서 **PR 제목과 본문을 자동 생성**한다.
생성 후 GitHub CLI(`gh`)로 PR을 바로 올리는 명령어까지 제공한다.

---

## 브랜치 전략 (GitHub Flow)

```
main                          ← 배포 브랜치 (항상 배포 가능 상태 유지)
├── feature/trip-form         ← 기능 개발
├── feature/google-maps       ← 기능 개발
├── fix/expense-bug           ← 버그 수정
└── chore/env-setup           ← 설정 변경
```

**핵심 규칙:**
- `main`이 유일한 베이스 브랜치
- 모든 작업은 `main`에서 브랜치를 파고 → PR → `main`으로 merge
- 브랜치 수명은 짧게 (1~3일 이내 merge 권장)
- merge 후 브랜치 즉시 삭제

### 브랜치 네이밍 규칙

| 브랜치 prefix | 용도 | PR 대상 |
|--------------|------|---------|
| `feature/`   | 새 기능 개발 | `main` |
| `fix/`       | 버그 수정 | `main` |
| `refactor/`  | 리팩토링 | `main` |
| `chore/`     | 설정, 패키지, 문서 | `main` |

---

## 사전 준비

PR 생성 요청이 들어오면 아래 정보를 먼저 파악한다.

```bash
# 현재 브랜치 확인
git branch --show-current

# main과의 변경 파일 목록 확인
git diff main --name-only

# 커밋 목록 확인
git log main..HEAD --oneline
```

파악한 정보를 바탕으로 PR 제목과 본문을 작성한다. 대상 브랜치는 항상 `main`.

---

## PR 제목 규칙

- **한국어**로 자유롭게 작성
- 이 PR에서 가장 핵심적인 변경 하나를 대표하는 문장
- 50자 이내
- 브랜치 이름을 그대로 옮기지 말 것

```
✅ 좋은 예
여행 생성 폼 및 Supabase 연동 구현
Google Maps 경로 시각화 기능 추가
지출 정산 계산 로직 및 멤버별 현황 UI 구현
항공편 3시간 전 도착 권유 알림 추가

❌ 나쁜 예
feat/trip-form 작업 완료       (브랜치 이름 그대로)
여러 가지 수정                  (무엇인지 불명확)
```

## PR 본문 템플릿

```markdown
## 개요
<!-- 이 PR에서 무엇을 왜 변경했는지 2~3문장으로 설명 -->

---

## 변경 사항
<!-- 주요 변경 내용을 항목별로 정리 -->

- 
- 
- 

---

## 테스트 방법
<!-- 리뷰어가 직접 테스트할 수 있도록 단계별로 작성 -->

1. 
2. 
3. 

---

## 스크린샷
<!-- UI 변경이 있는 경우 Before / After 스크린샷 첨부 -->
<!-- 변경 없으면 "해당 없음" 으로 작성 -->

| Before | After |
|--------|-------|
|        |       |

---

## 체크리스트

- [ ] 로컬에서 정상 동작 확인
- [ ] 콘솔 에러 없음
- [ ] 환경변수 누락 없음
- [ ] 불필요한 console.log 제거
- [ ] TypeScript 타입 오류 없음
- [ ] 모바일 화면 확인 (UI 변경 시)
```

---

## 변경 유형별 작성 가이드

### 새 기능 추가 (feat)
```
개요: 어떤 기능을 추가했고, 왜 필요한지
변경 사항: 추가된 컴포넌트, API 연동, DB 변경 등
테스트: 기능을 직접 사용해보는 시나리오 위주
```

### 버그 수정 (fix)
```
개요: 어떤 버그였는지, 어떻게 재현됐는지, 어떻게 수정했는지
변경 사항: 수정한 파일과 로직
테스트: 버그 재현 → 수정 후 정상 동작 확인 순서로
```

### 리팩토링 (refactor)
```
개요: 왜 리팩토링했는지 (성능, 가독성, 중복 제거 등)
변경 사항: 변경 전/후 구조 비교
테스트: 기존 기능이 그대로 동작하는지 확인
스크린샷: 해당 없음으로 작성
```

### DB 스키마 변경 (chore)
```
개요: 어떤 테이블/컬럼이 왜 변경됐는지
변경 사항: 변경된 SQL 내용 포함
테스트: Supabase에서 마이그레이션 적용 방법 명시
```

## 응답 형식

PR 내용 생성 시 아래 형식으로 응답한다.

````
## PR 제목
(제목)

## 대상 브랜치
(현재 브랜치) → main

## PR 본문
(본문 전체)

---
## GitHub CLI 명령어
```bash
gh pr create \
  --base main \
  --title "(제목)" \
  --body "$(cat << 'EOF'
(본문)
EOF
)"
```
````

## 주의사항
- 변경 내용 파악 없이 요청하면 `git diff`와 `git log`를 먼저 실행할 것
- 스크린샷 항목은 UI 변경이 없어도 템플릿에서 제거하지 말고 "해당 없음"으로 유지
- 체크리스트는 항상 전체 포함, 임의로 항목 삭제 금지
- PR 본문은 리뷰어 입장에서 읽기 쉽게 작성
